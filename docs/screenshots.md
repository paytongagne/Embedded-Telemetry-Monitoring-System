# Demo Captures

This page documents visual evidence from the local FastAPI demo. Captures should show successful API behavior, clear response bodies, and system-level functionality.

## Capture Set

| File | Endpoint or View | Purpose |
|---|---|---|
| `api-health.png` | `GET /health` | Confirms the service is running and returning version metadata |
| `api-devices.png` | `GET /api/v1/devices` | Shows device status summaries, alert counts, and latest health scores |
| `api-latest-telemetry.png` | `GET /api/v1/telemetry/latest` | Shows recent classified telemetry readings |
| `api-alerts.png` | `GET /api/v1/alerts` | Shows warning and critical alert records generated by classification logic |
| `api-resolve-alert.png` | `PATCH /api/v1/alerts/{alert_id}/resolve` | Shows the alert resolution workflow |
| `api-summary.png` | `GET /api/v1/summary` | Shows fleet-level totals for devices, alerts, and stored readings |
| `api-schema-contract.png` | OpenAPI schema panel | Shows request validation fields and accepted subsystem values |

## Asset Location

Place image files in:

```text
docs/assets/screenshots/
```

## Markdown Usage

Screenshots can be embedded in project documentation with relative paths:

```markdown
![Fleet summary endpoint](docs/assets/screenshots/api-summary.png)
![Device status endpoint](docs/assets/screenshots/api-devices.png)
```

## Capture Notes

Use captures that show successful `200` responses for primary examples. Validation errors can be useful while testing, but the main documentation should focus on the working API flow.

Avoid including unrelated browser tabs, tokens, local paths with sensitive names, or unrelated applications in captured images.
